# 追補指示書v2: 相場照合APIの502(クラッシュ/タイムアウト)を修正

対象: `agent/pr4-souba-check-ui` ブランチ(PR #5)および `api/market-price.js`。**同じPRのブランチ上で修正**すること。§0の絶対規則は引き続き適用。
本書は `CODEX_SPEC_phase1-3.md` §3.2の「直近8四半期を直列・1秒間隔で取得」という記述を**明示的に上書き**する。

## 症状(実測)
- SoubaCheck実行 → HTTP **502 Bad Gateway**、レスポンス `{"error":"国交省APIからデータを取得できませんでした"}`
- 502はVercelサーバーレス関数のクラッシュまたは実行時間超過(無料枠10秒)を示す

## 修正1: 時間予算の設計(502の直接対策・最優先)

- 関数冒頭で `const t0 = Date.now()` を取り、**全体8秒の時間予算**を管理する
- すべての外部fetchに **AbortControllerで個別タイムアウト4秒**を設定する
- XIT001の年次取得は**直列ではなく `Promise.all` による並列**で行う(ユーザー操作起点の少数リクエストのため許容。旧指示の「直列・1秒間隔」は本書で撤回)
- 予算超過時は取得済み分だけで集計を返すか、`{ok:false, stage:"timeout"}` を返す。**絶対に例外を未処理のまま投げない**(関数全体を try/catch で包み、catch では200で `{ok:false, stage, error}` を返す)

## 修正2: レスポンス読み取りとgzip解凍(一方通行で実装)

HTTPボディは一度しか読めない。以下の順で**一度だけ**読むこと(res.json()を先に試す実装は禁止):

```js
const buf = Buffer.from(await res.arrayBuffer());   // 常にバイト列で一度だけ読む
let text;
if (buf[0] === 0x1f && buf[1] === 0x8b) {           // gzipマジックナンバー
  text = zlib.gunzipSync(buf).toString("utf8");     // import zlib from "zlib"
} else {
  text = buf.toString("utf8");
}
const json = JSON.parse(text);                       // 失敗はcatchでstage:"parse"
```

リクエストヘッダには `Ocp-Apim-Subscription-Key` に加えて `Accept-Encoding: gzip` を付与してよい。

## 修正3: 取得パラメータと対象年

- **XIT002**(市区町村一覧): `?area=<都道府県コード2桁>`。返却要素のコード/名称はフィールド名の揺れに備え、`id`/`code` と `name` のいずれでも拾えるよう防御的にマッピング。結果はモジュールスコープでメモリキャッシュ(ベストエフォート)
- **XIT001**(取引価格): `?year=<西暦4桁>&city=<市区町村コード5桁>`。`quarter`は使わない
- **対象年**: 直近年はデータ未整備のため使わない。**現在年-2 を起点に過去3年分**(2026年なら 2024, 2023, 2022)を並列取得し結合。`data` が無い/空の年はスキップ
- 種別フィルタ: `Type` を部分一致で判定 — mansion→「中古マンション」を含む / house→「宅地(土地と建物)」を含む / land→「宅地(土地)」を含む
- 単価: `TradePrice / Area`。両方が正の数値の行のみ。外れ値(1万円/㎡未満、1,000万円/㎡超)は除外

## 修正4: エラーの可視化

- 返却: 成功 `{ok:true, count, medianUnitYenPerM2, p25, p75, samples(≤50), city:{code,name}, years:[取得に使った年]}` / 失敗 `{ok:false, stage, error}`
- `stage` ∈ `"auth"`(MLITが401/403) / `"xit002"`(市区町村解決失敗) / `"empty"`(全年0件) / `"parse"` / `"timeout"` / `"other"`
- **HTTPステータスは常に200で返す**(クライアント側でokフラグ判定。502や500を返さない)
- サーバーの `console.log` に: 解決した市区町村コード、各年の取得件数、失敗時はレスポンス先頭300文字(Vercelログで確認可能に)
- クライアント(souba.jsx)は stage に応じた日本語表示:
  - auth → 「APIキーが未設定または無効です(Vercelの環境変数 MLIT_API_KEY を確認してください)」
  - xit002 → 「市区町村が見つかりませんでした。表記(例: 文京区)をご確認ください」
  - empty → 「この地域・種別の取引データが見つかりませんでした。種別や市区町村を変えてお試しください」
  - timeout → 「データ取得に時間がかかっています。時間をおいて再度お試しください」
  - parse/other → 「データの取得に失敗しました。時間をおいて再度お試しください」

## 修正5: UI文言の整合

SoubaCheckの説明文「直近2年の成約・取引事例から〜」を「**直近の取引事例(データ整備済みの過去3年分)から〜**」に変更し、結果表示にも取得対象年(`years`)を小さく併記する。

## 変更しない点
